- 严格来说，目前测试的并发脚本都不能并行下载，使用 `concurrent.futures` 库实现的示例受到 GIL (Global Interpreter Lock，全局解释器锁)的限制，而用于下载的 .py 文件只能单线程运行。

- 但是 GIL 几乎对 I/O 密集型处理无害。

#### 阻塞型 I/O 和 GIL

- CPython 解释器不是线程安全的。有全局解释器锁，一次只允许使用一个县城执行 Python 字节码。因此通常不能使用多个 CPU 核心。

- 编写代码的时候无法控制；执行耗时任务的时候可以通过 C 语言拓展或内置函数释放 GIL。有一个对应的 C 语言的 Python 库，可以自行启动系统线程，利用可用 CPU 核心，但是大量增加代码复杂度。

- 标准库中执行阻塞型 I/O 操作的函数，等待操作系统返回结果的时候都会释放 GIL，因此这个层次上可以多线程，I/O 密集型 Python 也就从中受益。
